## 1. 專案概述 (Project Overview)

* **專案名稱：** 《阿晴 Dialogues》獨立 Podcast 網站（MVP）
* **目標受眾：** 由 Instagram Reels / 貼文導流，尋求情感共鳴與理性思考醫療／生命議題的聽眾。
* **核心目的：**
  1. 建立品牌自有網域與 SEO 資產（不依賴平台演算法）。
  2. 提供 Web 端流暢的非同步播放體驗（含時間戳跳轉）。
* **明確不做（Out of Scope, MVP）：** 會員系統、後台 CMS、逐字稿全文搜尋、深色模式切換、**匿名留言板／悄悄話牆（v1.3 決議整個移出 MVP——公開匿名留言必然伴隨審核人力與後端維運，與「零維運」的架構初衷衝突；聽眾回饋暫由 IG DM 承接，未來若要重啟建議以匿名表單私收、不公開上牆的形式評估）**。

## 3. 功能規格 (Functional Specs)

### 功能一：RSS Feed 自動解析器 (RSS Parser)

**說明：** 網站無後台。每次 Build 時 fetch 遠端 RSS 並解析為資料。

**v1.1 修正重點：**

1. **Slug 改用 GUID，不用標題。** 標題事後改字（改錯字、加集數）會導致 URL 變動 → 舊分享連結 404、SEO 權重歸零。RSS 的 `<guid>` 為永久識別碼，slug 規則：`ep-{guid 短雜湊}` 或若 GUID 本身可讀則直接 slugify；標題僅作顯示用。
2. **改用 `rss-parser`。** 自動處理 `itunes:*` 欄位與 CDATA，不需手動處理 `explicitArray` 陷阱。
3. **Build 容錯。** RSS fetch 失敗或格式異常時，Build 應 fail loudly（讓 Vercel 保留上一版部署），而非產出空站。個別欄位缺漏（如缺 `itunes:duration`）給預設值，不中斷整批解析。

```javascript
// src/utils/rss.js（build-time only，不進 client bundle）
import Parser from 'rss-parser';

const parser = new Parser({
  customFields: { item: [['itunes:duration', 'duration'], ['itunes:image', 'itunesImage']] }
});

export async function getEpisodes(rssUrl) {
  const feed = await parser.parseURL(rssUrl); // 失敗直接 throw，讓 build 中斷
  return feed.items.map((item) => ({
    guid: item.guid,
    slug: toStableSlug(item.guid),          // ⚠️ 以 GUID 為準，非 title
    title: item.title ?? '未命名',
    descriptionHtml: item.content ?? '',     // Show notes（含 HTML）
    pubDate: new Date(item.pubDate ?? Date.now()),
    audioUrl: item.enclosure?.url ?? null,   // 缺音檔 → 頁面隱藏播放器並記 log
    duration: item.duration ?? null,
    cover: item.itunesImage?.$?.href ?? feed.image?.url
  }));
}
```

**驗收標準（AC）：**
- 修改任一集標題後重建，該集 URL 不變。
- RSS 端點回 500 時，線上網站維持前一版內容。
- 缺 `enclosure` 的頁面不噴錯，僅隱藏播放器。

### 功能二：時間戳點擊跳轉播放 (Timestamp Audio Sync)

**說明：** 頁內建 HTML5 `<audio>`。Show notes 中的 `MM:SS` / `HH:MM:SS` 文字轉為可點擊元素，點擊後跳轉播放。

**v1.1 修正重點：**

1. **Build-time 轉換，非 runtime 掃描。** 在 Astro build 階段就用 regex 把時間戳包成 `<button>`，client 只掛一個事件委派 listener，減少 runtime 工作與 CLS。
2. **Regex 明確定義：** `/\b(\d{1,2}:)?\d{1,2}:\d{2}\b/g`，並排除已在 `<a>` 內的文字（避免破壞原有連結）。
3. **無障礙：** 用 `<button class="timestamp-link" data-time="04:20" aria-label="跳轉至 4 分 20 秒">04:20</button>`，而非 `<a href="#">`（鍵盤可操作、不污染 history）。
4. **邊界行為：** 秒數超過音檔長度 → clamp 至結尾；`audio` metadata 尚未載入 → 先 `load()` 再於 `loadedmetadata` 後 seek。

```javascript
// 事件委派（頁唯一 client script）
document.addEventListener('click', (e) => {
  const btn = e.target.closest('.timestamp-link');
  if (!btn) return;
  const audio = document.getElementById('main-audio-player');
  const seconds = toSeconds(btn.dataset.time);
  const seek = () => {
    audio.currentTime = Math.min(seconds, audio.duration || seconds);
    audio.play();
  };
  audio.readyState >= 1 ? seek() : audio.addEventListener('loadedmetadata', seek, { once: true });
});

function toSeconds(t) {
  return t.split(':').map(Number).reduce((acc, n) => acc * 60 + n, 0);
}
```

**AC：**
- `04:20`、`1:04:20` 皆可點擊並正確跳轉。
- Show notes 內原有的外部連結不被時間戳轉換破壞。
- Tab 鍵可聚焦時間戳並以 Enter 觸發。

### 功能三：一鍵跳轉智慧連結 (Smart Platform Links)

**v1.1 修正重點——策略反轉：** 原設計以 custom scheme（`podcast://`、`spotify:`）為主、https 為備援；實務上 custom scheme 在未安裝 App 時會**靜默失敗**（無任何反應）。正確做法：

1. **一律輸出標準 `https://` Universal Link / App Link**（`https://podcasts.apple.com/...`、`https://open.spotify.com/show/...`）。已安裝 App 的手機會由 OS 自動接手開啟 App；未安裝則優雅降級到網頁。
2. 不做 UA 偵測、不寫跳轉 JS——零維護成本且行為最可預期。
3. 按鈕位置：首頁 hero 區 + 每個頁播放器下方，含 Apple Podcasts / Spotify 官方 badge。

## 4. 自動化部署流程 (CI/CD Pipeline)

**v1.1 修正：** 不比對 `lastBuildDate`（部分託管平台每次請求都回新值，或改 metadata 不更新它）。改比對**最新一集的 `<guid>` 或 feed 內容雜湊**。

1. **輪詢（預設方案）：** GitHub Actions cron 每 6 小時執行：fetch RSS → 取最新 guid → 與 repo 內快取檔比對 → 有差異則 commit 快取檔並呼叫 **Vercel Deploy Hook**。
2. **Webhook（快速路徑）：** 需確認 SoundOn 後台是否提供上架 Webhook / Zapier 整合；若有，上架時直接打 Vercel Deploy Hook，5 分鐘內上線。無論有無，cron 輪詢皆保留作為保底。
3. **失敗通知：** Build 失敗時 GitHub Actions 寄信 / Discord webhook 通知。

## 5. MVP 頁面架構 (Information Architecture)

* **`/`（首頁）：** 品牌主視覺與簡介、最新 3 集卡片（標題、日期、摘要、播放鈕）、平台訂閱 badge。
* **`/episodes`：** 全部列表。MVP 用**靜態分頁**（每頁 10 集）而非無限滾動——SSG 免 API、對 SEO 與分享連結友善。
* **`/episodes/[slug]`：** 標題、封面、播放器、格式化 Show notes（含可點擊時間戳）、底部平台訂閱 badge + IG 連結（承接原留言板的聽眾回饋動線）。
* **`/404`：** 自訂 404，導回列表（承接舊連結失效情境）。

### 5.1 品牌視覺規範 (Brand & Design Tokens) — v1.2 新增

視覺定調已由主視覺 Logo（《A. Chin. Logs》手寫泡泡字，同時作為 Spotify 節目封面）拍板：**溫暖極簡、手感、單一品牌藍**。網站所有 UI 由此 Logo 延伸，色票直接取自原檔：

| Token | 值 | 用途 |
|---|---|---|
| `brand.blue` | `#2D56A8` | 主色：標題、播放鈕、時間戳 hover、連結 |
| `brand.cream` | `#EEEDE9` | 全站背景（取代純白，維持 Logo 的暖灰底） |
| `brand.blue-dark` | `#1F3D7A`（衍生） | hover / active 狀態 |
| `brand.blue-tint` | `#2D56A8` @ 8–12% | 卡片底色、時間戳 pill 底 |
| 文字色 | `#2A2A28`（近黑暖灰） | 內文；標題可直接用 `brand.blue` |

**設計原則：**
1. **雙色系統**：整站僅藍 + 米白兩色系（含其濃淡階），不引入第三色；強調用留白與字重，不用新顏色。
2. **字體**：標題選圓潤無襯線（如 Baloo 2 / Fredoka 呼應泡泡字手感；中文搭 Noto Sans TC Medium）；內文 Noto Sans TC Regular。Logo 本體為圖片使用，不嘗試用 webfont 重現。
3. **形狀語言**：大圓角（`rounded-2xl` 起跳）、按鈕與時間戳做成 pill 形，呼應 Logo 筆畫的圓潤端點。
4. **對比度**：`#2D56A8` on `#EEEDE9` 對比約 6.3:1，內文與 UI 元件均過 WCAG AA；小於 14px 的次要文字避免用 blue-tint 當前景色。

**平台封面規格（同一 Logo 檔沿用）：**
* 現有原檔 2000×2000 JPG — ✅ 已符合 Apple Podcasts（1400–3000px、RGB、方形）與 Spotify（≥640×640、方形）要求，可直接沿用，無需重輸出。
* 建議另備 3000×3000 母檔以備未來平台調升規格；網站用途另輸出 512px WebP（首頁 hero）與 180px（favicon / apple-touch-icon）。
* OG 分享圖（1200×630）：米白底置中 Logo + 標題文字，可由 Astro 以模板批次生成（如 `@vercel/og` 或 satori）。

## 6. 非功能性需求 (NFR)

| 項目 | 目標 |
|---|---|
| 效能 | Lighthouse Performance ≥ 90（行動裝置）；首頁 JS < 50KB gzip |
| 無障礙 | 播放器與時間戳可全鍵盤操作；對比度符合 WCAG AA |
| 分析 | 隱私友善方案（Vercel Analytics 或 Plausible），不放第三方廣告追蹤 |
| 相容性 | iOS Safari / Android Chrome 近兩版；音檔跨域播放需確認託管平台 CDN 的 CORS 與 `Range` request 支援 |
